# LogisticsMod — Changes 2026-05-12

Fixes the planner-race that caused `[LOGI]` cycles to be cleaned up and re-dispatched while the async planner was still working on them, fixes the sticky "in transit" UI status, routes all user-facing strings through `LEManager.Get` so translation packs can override them, and makes GET-row text wrap so long status notes no longer clip. Plus dead-code cleanup and two small correctness fixes.

## 1. Planner-race fix (the main bug)

**Symptom:** under heavy time acceleration, freshly-dispatched cycles were torn down as "stale" by `HasActiveCycleDelivering` before Unity's CodeJobSystem callback (`GameManager.SetPMParameterForCodeJobSystem`) had a chance to flip `cmd.wasSetPMParameterForCodeJobSystem = true`. Once both the `_cycleCreatedAt` grace and the `_pendingPlanningDeliveries` grace expired, `OnDayChange` would `RemoveCycleMission(cmd)` and re-dispatch on the same tick. Reproducible by setting the grace to 0 and running a forced relay route.

**Fix** — `Logic/LogisticsObserver.cs`, inside `IsCycleWaitingOrPlanned`:

For each spacecraft attached to the cycle, if it is still tracked in `_returnHomeByShipId` with `HasLeftHome == false`, the cycle is treated as planned-in-progress regardless of how much in-game time has passed. The carrier provably hasn't departed, so the planner can't yet have driven the ship to its destination, so cleanup must wait.

```csharp
if (_returnHomeByShipId.TryGetValue(sc.ID, out var returnState)
    && returnState != null
    && !returnState.HasLeftHome)
{
    return true;
}
```

This makes the grace window a soft fallback rather than the only line of defence.

## 2. Two new config entries — `[Diagnostics]` section

Moved from hardcoded constants in `LogisticsObserver.cs` to `Plugin.cs` config bindings, so they can be tuned without rebuilding:

| Key | Default | Replaces |
|-----|---------|----------|
| `CyclePlanningGraceDays` | `3.0` | `private const double NewCyclePlanningGraceDays = 3` |
| `VerboseLogging` | `false` | `private static readonly bool VerboseLogging = false` |

Set `CyclePlanningGraceDays = 0.0` to verify the race-guard fix above — cycles will now stay alive purely on the carrier-still-home check.

## 3. `ClampToOutstandingRequest` now subtracts in-flight

`Logic/LogisticsObserver.cs`. Previously the function returned `target − currentStock`, which double-counted any in-flight deliveries when the relay paths re-entered `SetupDirectCycleMission` mid-OnDayChange. Now subtracts `GetInFlightDeliveryAmount` as well, matching the bookkeeping `OnDayChange` already performs at the top of the loop.

## 4. Sticky "in transit" status fix

**Symptom:** the GET-section row shows "[in transit]" forever after a cycle has completed and there is no inbound vehicle, especially on multi-trip refills (`useMinimumAmount=true`) when one trip completes but stock hasn't reached `requestedAmount`.

**Root cause:** the only line in `OnDayChange` that touched `req.status` after `TryCreateDeliveries` was:

```csharp
if (req.status == Pending && !string.IsNullOrEmpty(pendingReason))
    req.statusNote = pendingReason;
```

This *reads* `Pending` but never *writes* it. So once a request had reached `InProgress` on any earlier tick (via `SetupDirectCycleMission` etc.), and the delivering cycle later completed without filling to target, the next tick's path past `HasActiveCycleDelivering`/`HasPendingPlanningDelivery`/`inFlight`/relay-progress into `TryCreateDeliveries` would *not* clear the stale `InProgress` if dispatch failed — leaving the UI saying "in transit" with nothing actually shipping.

**Fix** — `Logic/LogisticsObserver.cs`, immediately before `TryCreateDeliveries`:

```csharp
req.status = Data.LogisticsRequestStatus.Pending;
var pendingReason = TryCreateDeliveries(req, requesterOI, rd, remaining, player);
```

If `TryCreateDeliveries` dispatches, the `SetupDirectCycleMission` / `SetupCycleMission` / `ExecuteRouteCandidate` paths re-promote to `InProgress` as before. If dispatch fails, the request correctly reads `Pending` with `statusNote = pendingReason` (e.g., "No surplus X available at Y") and the UI shows the right state. All the prior `continue` branches that legitimately want to keep `InProgress` (active cycle, pending plan, in-flight, blocked return, relay progress) bypass this line, so behavior there is unchanged.

## 5. Localized user-visible strings

All status words and `statusNote` / planner blocker reasons now flow through a new helper, `Logic/LogisticsStrings.cs`. Every visible message goes through `LEManager.Get("logisticsmod.<key>", "<english fallback>")`, matching the localization pattern the codebase already uses for resource names (`LEManager.Get(rd.ID, rd.ID)` at `UI/LogisticsUI.cs:1243`).

If a translation pack supplies the keyed values, the UI picks them up automatically. If not, the English fallback (identical to the prior hardcoded text) is used, so out-of-the-box behavior is unchanged.

Keys are namespaced under three buckets:

- `logisticsmod.status.*` — the four request states (pending / in_transit / satisfied / failed).
- `logisticsmod.note.*` — relay-progress and return-blocked messages (staging_to, staged_at, shipping_from, waiting_spacecraft, waiting_lv, awaiting_return, waiting_return_fuel, return_blocked_on_ship, no_provider).
- `logisticsmod.blocker.*` — planner reasons surfaced when `TryCreateDeliveries` fails (no_surplus, no_sc_quota, no_sc_present, no_sc_matching, sc_quota_full, no_sc_idle, no_sc_available, no_provider_selected, no_surface_launch, no_lv_quota, no_lv_ready, no_lv_matching, lv_quota_full, no_lv_available, no_container, no_cargo_capacity, no_orbital_payload_capacity, no_source_orbit, no_logistics_data).
- `logisticsmod.transit.*` — UI suffix when an inbound vehicle is known (on_vehicle, arrives_only, on_vehicle_arrives).

Templates use `{0}`/`{1}` positional placeholders so translations can reorder words around the resource / object name.

## 6. GET-row text wraps instead of clipping

`UI/LogisticsUI.cs`. Now that `statusNote` is populated reliably (see fix #4), long pending reasons like "No matching launch vehicle quota at <body> [available launch support=...]" would clip off the right edge of the row. Two changes fix that without touching other sections:

- `MakeHLRow` now uses `LayoutElement.minHeight` plus a `ContentSizeFitter(verticalFit = PreferredSize)` instead of a fixed `preferredHeight`. Short rows still settle at the original height; rows whose label wraps grow to accommodate it. Other sections see no visual change because their labels are single-line.
- The GET request label gets `enableWordWrapping = true`, `overflowMode = Overflow`, plus an explicit `LayoutElement` with `flexibleWidth = 1f, preferredWidth = 0f`. Without that LayoutElement, TMP reports its full unwrapped text width to the `HorizontalLayoutGroup`, which would let the row sprawl horizontally instead of wrapping. With it, the label is forced to shrink to whatever width the row gives it and wraps cleanly.

The X-button on the right keeps its 24f preferred width and stays aligned to the top-right because the row's `childAlignment` is `MiddleLeft`. A tall wrapped row leaves the button at the top of its allotted height, which reads correctly.

## 7. Dead-code removal

All confirmed unused via `rg`:

- `LogisticsRequest.takeFuelFromTarget` — field was written by `AddRequest` and never read.
- `LogisticsRequest.homeProvider` — declared, never assigned or read.
- `LogisticsRequest.activeMissionId` — declared with default `-1`, never assigned or read.
- `LogisticsObserver.ResolveReturnHome(...)` — method defined, never called.
- `LogisticsNetwork.AddRequest` overloads lost the `takeFuelFromTarget = false` parameter. The only external caller — `UI/LogisticsUI.cs:1133` — uses the 5-arg form that didn't pass it anyway.

Nothing in `LogisticsPersistence` referenced any of these fields, so save-file compatibility is preserved.
